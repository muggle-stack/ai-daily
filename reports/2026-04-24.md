### 用"鹈鹕测试"通过半官方 Codex 后门 API 评测 GPT-5.5（A pelican for GPT-5.5 via the semi-official Codex backdoor API）
Simon Willison 在 GPT-5.5 发布当天分享了他的第一手体验。他认为这是一个快速、高效且能力极强的模型——"让它造什么就造什么"，但很难用语言精确描述它好在哪里。由于 API 尚未开放，Willison 巧妙地通过 OpenAI Codex 中暴露的接口（他称之为"OpenClaw 后门"）来运行他标志性的"鹈鹕基准测试"，以避免 ChatGPT 隐藏系统提示对结果的干扰。这篇文章既是一篇实用的模型评测，也揭示了一个有趣的现象：在 API 正式上线前，开发者社区已经在寻找各种方式来"裸测"新模型。对于关注模型真实能力而非包装效果的开发者来说，这种方法论本身就值得参考。

## 📄 研究论文

### 加速面向科研的可靠代码开发（On Accelerating Grounded Code Development for Research）
这篇论文直击一个痛点：AI 编程智能体在材料科学、通信工程、生物工程等细分科研领域表现不佳，因为基础模型缺乏最新的领域专业知识。研究者既没有资源微调大模型，也无法持续嵌入新发现，这成了科研人员采用 AI 编程工具的主要障碍。论文提出了一个新框架，旨在让编程智能体能够利用不断演进的领域知识来生成可靠代码。这项工作对于推动 AI 辅助编程从通用场景走向专业科研场景具有重要意义，尤其是对那些探索新型化合物、设计新协议的一线研究者。

🔗 [原文链接](https://arxiv.org/abs/2604.19022) · 来源：arXiv AI · 关键词：#代码智能体 #科研编程 #领域知识

### ARES：策略-奖励系统的自适应红队测试与端到端修复（ARES: Adaptive Red-Teaming and End-to-End Repair of Policy-Reward System）
RLHF 是当前大模型对齐的核心手段，但奖励模型（RM）一旦出错就会成为单点故障。现有红队测试主要针对策略层面的弱点，忽略了一种更危险的情况：核心 LLM 和奖励模型同时失效的"系统性漏洞"。ARES 框架正是为发现和修复这类双重漏洞而设计的，它引入"安全导师"角色，通过动态组合话题、角色、策略和目标来生成语义连贯的对抗性提示。这种从系统层面而非单点层面思考安全的视角，对构建更健壮的对齐流程很有启发。

🔗 [原文链接](https://arxiv.org/abs/2604.18789) · 来源：arXiv AI · 关键词：#RLHF #红队测试 #奖励模型 #安全对齐

### 推理结构决定推理模型的安全对齐效果（Reasoning Structure Matters for Safety Alignment of Reasoning Models）
大型推理模型（LRM）在复杂推理任务上表现出色，但面对恶意查询时常常"翻车"。这篇论文发现问题的根源在于推理结构本身，并据此提出了 AltTrain 方法——通过改变推理结构来实现安全对齐。AltTrain 的亮点在于极其轻量：不需要复杂的强化学习训练或奖励设计，仅用 1K 条训练样本的监督微调（SFT）即可完成。实验在多个 LRM 骨干网络上验证了其有效性和泛化能力，为推理模型的安全对齐提供了一条低成本、高实用性的路径。

🔗 [原文链接](https://arxiv.org/abs/2604.18946) · 来源：arXiv AI · 关键词：#推理模型 #安全对齐 #AltTrain #推理结构

### OLLM：基于选项的大语言模型（OLLM: Options-based Large Language Models）
标准 LLM 的下一个 token 预测是单一路径的，OLLM 提出用一组"可学习选项"来替代，通过离散潜变量索引多个合理的下一 token 候选。与依赖温度或采样策略来引入多样性不同，OLLM 显式地建模了变异性。架构上它是一个轻量级"插件"，只需在输出头前插入编码器和解码器两层，就能将几乎任何预训练 LLM 转换过来。在 1.7B 参数的骨干网络上实验，仅需训练 1.56% 的参数。这种思路为提升 LLM 生成多样性和可控性提供了一个优雅的结构化方案。

🔗 [原文链接](https://arxiv.org/abs/2604.19087) · 来源：arXiv AI · 关键词：#OLLM #next-token预测 #潜变量 #LLM架构

## 🔧 开源项目

### 我们用 7000+ 次调用对 18 个 LLM 做了 OCR 基准测试——便宜的老模型经常赢（We benchmarked 18 LLMs on OCR (7k+ calls) — cheaper/old models oftentimes win）
团队发现很多人在 OCR 场景上严重多花钱，于是精选了 42 份标准文档，对 18 个模型各跑 10 次，共计 7,560 次调用。核心结论：对于标准 OCR 任务，较小和较老的模型在准确率上能匹配旗舰模型，而成本只是零头。评测维度包括 pass^n（规模化可靠性）、每次成功成本、延迟和关键字段准确率。整套基准测试框架和数据集已在 GitHub 开源，还提供了免费工具供用户测试自己的文档。对于正在搭建文档提取流水线的团队来说，这是一份非常实用的选型参考。

### LiteParse 浏览器版：在浏览器中提取 PDF 文本（Extract PDF text in your browser with LiteParse for the web）
LlamaIndex 的开源项目 LiteParse 原本是一个 Node.js CLI 工具，用于从 PDF 中提取文本。Simon Willison 将其移植到了浏览器端运行，复用了大部分原有库。LiteParse 的特色在于不依赖 AI 模型，而是用传统 PDF 解析加 Tesseract OCR 兜底，核心难点是通过"空间文本解析"的巧妙启发式算法来处理多栏布局等复杂排版，以正确顺序提取文本。浏览器版意味着用户无需后端服务即可完成 PDF 文本提取，对隐私敏感场景和轻量级应用尤其友好。

### FP16 + ONNX 之后如何继续优化 Transformer 模型大小与推理速度（Optimizing Transformer model size & inference beyond FP16 + ONNX）
一位开发者分享了在 Transformer 模型优化上遇到的瓶颈：已经完成 FP16 转换（约 2 倍体积缩减）和 ONNX Runtime 导出，尝试了非结构化/结构化剪枝和 ONNX 图优化，但模型仍停留在约 162MB，收益甚微。社区讨论了几个进阶方向：低秩分解（SVD/LoRA 风格压缩）、更激进的量化（INT8/INT4，如 GPTQ、AWQ、SmoothQuant）、知识蒸馏到更小的学生模型，以及硬件特定优化如 TensorRT 和 FlashAttention。对于同样在模型压缩上"卡关"的工程师来说，这个讨论串提供了一份清晰的进阶路线图。
